# 病虫害预警功能详解

> **文档用途**：本文件为 AgroGuard（草地作物病虫害智能监测系统）毕业设计论文的核心技术章节，详细阐述病虫害预警模块的完整实现原理、前后端交互流程、核心算法及数据库设计。

---

## 一、功能概述

病虫害预警模块是 AgroGuard 系统的**核心功能**之一，其设计目标是对草地作物进行**全生命周期的病虫害防护监测**，自动计算每株草的下一次防护截止日期，并在防护超期时向管理员发出预警提醒。

该模块涉及以下核心实体：
- **草地基础信息**（`basic_basicinfo`）：每株草的编号、品种、出生日期、所在站点等
- **防护记录**（`d_plantcare_immunizationinfo`）：历史施药/防护记录
- **防护剂信息**（`supply_commodityinfo`）：可用的防护剂名称、说明、类型
- **阈值配置**（`threshold_setting`）：每种防护剂的防护周期（天数）
- **预警信息**（`w_information_immunizationMessageinfo`）：系统生成的预警记录

### 2.4 阈值设置弹窗（thresholdSet.vue）

阈值设置是一个 `el-dialog` 弹窗，用于配置每种防护剂的防护周期和防护模式。

**核心配置项**：
- **日间阈值（day）**：滑块组件，范围 0~365 天，表示防护周期
- **防护模式**：
  - `ifyear = 0`：循环防护（到期后可再次防护）
  - `ifyear = 1`：全周期仅防护一次（终身只需防护一次）

**显示名称映射**（IMM_CNAME_DISPLAY_MAP）：

后端返回的防护剂名称（如"草地锈病防治剂"）在前端通过映射表转换为更简洁的显示名称（如"锈病防治剂"），以提升界面可读性。

```python
def get_thresholdsetMessageinfo():
    query = ThresholdSetMessageinfo.query
    total = query.count()
    list = []
    for info in query:
```

### 3.5 阈值编辑接口

**路由**：`POST /w_information/thresholdsetMessageinfo/edit`

**功能**：批量修改阈值配置。当 `ifyear` 发生变化时，自动将 `ifchange` 置为 `1`。

```python
def edit_thresholdsetMessageinfo():
    data = request.get_json()
    for i in range(len(data)):
        if "cname" in data[i]:
            del data[i]["cname"]  # 删除前端拼接的 cname，避免写入阈值表
```

### 3.6 全量更新预警接口（核心算法）

**路由**：`POST /w_information/thresholdsetMessageinfo/update`

**功能**：手动触发全量预警数据更新。该接口是病虫害预警模块的**核心计算引擎**，通过 6 条 SQL 语句完成完整的预警数据生命周期管理。

#### 3.6.1 更新流程总览

```
开始
  │
  ├── Step 0: 查询 threshold_setting，获取所有防护剂配置
  │     ├── ifchange == 1 → 加入 list_init（需要初始化）
  │     └── ifchange == 0 → 加入 list_normal（仅需日常更新）
  │
  ├── Step 1 (初始化): 对 list_init 中的每种防护剂
  │     ├── query11: 删除该防护剂的所有旧预警记录
  │     ├── query01: 从 threshold_setting 读取 threshold 到变量 @vaccine_threshold
  │     ├── query1 (有防护记录): 为已有防护记录的草株生成预警（基于 MAX(imm_date)）
  │     ├── query2 (无防护记录): 为从未防护的草株生成预警（基于 birth）
  │     └── query12: 将该防护剂的 ifchange 置为 0
  │
  ├── Step 2 (日常更新): 对所有防护剂执行
  │     ├── query3: 同步最新防护记录（imm_date、dead_date、mon_age）
  │     ├── query4: 同步草株基础信息（house、hurdle_name、mon_age）
  │     ├── query5: 删除已淘汰/死亡草株的预警记录
  │     └── query6: 删除"仅防护一次"且已防护过的记录
  │
  └── 提交事务
```

#### 3.6.2 核心 SQL 详解

**query11 — 清空旧预警数据**：

```sql
DELETE w
FROM w_information_immunizationMessageinfo AS w
WHERE w.vaccine_id = :id;
```

> 在重新初始化某防护剂前，先删除该防护剂的所有历史预警记录，避免数据重复。

**query01 — 读取防护周期阈值**：

```sql
SELECT threshold INTO @vaccine_threshold
FROM threshold_setting
WHERE vaccine_id = :id;
```

> 将 `threshold_setting` 中该防护剂的 `threshold` 值读取到 MySQL 用户变量中，供后续 SQL 使用。

**query1 — 为已有防护记录的草株生成预警**：

```sql
INSERT into w_information_immunizationMessageinfo
(basic_id, ele_num, pre_num, sex, variety, mon_age, house, hurdle_name,
 vaccine_id, birth, imm_date, distance_date, dead_date, state, f_staff, f_date, note)
SELECT
  d.basic_id AS basic_id,
  b.ele_num AS ele_num,
  b.pre_num AS pre_num,
  b.sex AS sex,
  b.variety AS variety,
  ROUND((DATEDIFF(NOW(), b.birth) / 30), 1) AS mon_age,
  b.house_name AS house_name,
  b.hurdle_name AS hurdle_name,
  d.vaccine_id AS vaccine_id,
  b.birth AS birth,
  d.max_immdate AS imm_date,
  DATEDIFF(NOW(), d.max_immdate) AS distance_date,
  DATE_ADD(d.max_immdate, INTERVAL @vaccine_threshold day) AS dead_date,
  1 AS state,
  'System' AS f_staff,
  NOW() AS f_date,
  COALESCE(s.explain, '') AS note
FROM basic_basicinfo AS b
LEFT JOIN (
  SELECT basic_id, vaccine_id, MAX(imm_date) AS max_immdate
  FROM d_plantcare_immunizationinfo
  WHERE vaccine_id = :id
  GROUP BY basic_id
) AS d ON b.id = d.basic_id
LEFT JOIN supply_commodityinfo s ON s.id = d.vaccine_id
WHERE d.vaccine_id = :id
  AND b.state = 1
  AND b.variety IN (0, 1)
ORDER BY b.id ASC;
```

**关键计算逻辑**：
- `mon_age = ROUND(DATEDIFF(NOW(), b.birth) / 30, 1)`：当前日期与出生日期的天数差除以30，得到生长月数
- `distance_date = DATEDIFF(NOW(), d.max_immdate)`：当前日期与最近一次防护日期的天数差
- `dead_date = DATE_ADD(d.max_immdate, INTERVAL @vaccine_threshold day)`：**最近一次防护日期 + 防护周期 = 下次防护截止日期**
- `COALESCE(s.explain, '')`：从防护剂信息表获取效果说明，为空时返回空字符串

**query2 — 为从未防护的草株生成预警**：

```sql
INSERT into w_information_immunizationMessageinfo
(basic_id, ele_num, pre_num, sex, variety, mon_age, house, hurdle_name,
 vaccine_id, birth, imm_date, distance_date, dead_date, state, f_staff, f_date, note)
SELECT
  b.id,
  b.ele_num AS ele_num,
  b.pre_num AS pre_num,
  b.sex AS sex,
  b.variety AS variety,
  ROUND((DATEDIFF(NOW(), b.birth) / 30), 1) AS mon_age,
  b.house_name AS house_name,
  b.hurdle_name AS hurdle_name,
  :id AS vaccine_id,
  b.birth AS birth,
  b.birth AS imm_date,                          -- 无防护记录时，imm_date = birth
  DATEDIFF(NOW(), b.birth) AS distance_date,    -- 从出生到现在已过去的天数
  DATE_ADD(b.birth, INTERVAL @vaccine_threshold day) AS dead_date,
  1 AS state,
  'System' AS f_staff,
  NOW() AS f_date,
  COALESCE(s.explain, '') AS note
FROM basic_basicinfo AS b
LEFT JOIN supply_commodityinfo s ON s.id = :id
WHERE b.state = 1
  AND b.variety IN (1, 0)
  AND b.id NOT IN (SELECT basic_id FROM d_plantcare_immunizationinfo);
```

> 对于从未施药的草株，假设其"上次防护日期"为出生日期（`imm_date = birth`），截止日期为 `birth + threshold`。

**query3 — 同步最新防护记录**：

```sql
UPDATE w_information_immunizationMessageinfo w
INNER JOIN d_plantcare_immunizationinfo d
  ON w.basic_id = d.basic_id AND w.vaccine_id = d.vaccine_id
INNER JOIN threshold_setting t
  ON t.vaccine_id = w.vaccine_id
SET w.imm_date = d.imm_date,
    w.mon_age = ROUND((DATEDIFF(CURRENT_DATE, w.birth) / 30), 1),
    w.distance_date = DATEDIFF(CURRENT_DATE, d.imm_date),
    w.dead_date = DATE_ADD(d.imm_date, INTERVAL threshold day),
    w.f_date = NOW()
WHERE w.imm_date < d.imm_date
  AND t.ifyear = 0;
```

> 当 `d_plantcare_immunizationinfo` 中新增了更晚的防护记录时，同步更新预警表中的 `imm_date` 和 `dead_date`。仅对**循环防护**（`ifyear = 0`）的防护剂生效。

**query4 — 同步草株基础信息**：

```sql
UPDATE w_information_immunizationMessageinfo AS w
INNER JOIN basic_basicinfo b ON b.id = w.basic_id
SET w.mon_age = ROUND(DATEDIFF(CURRENT_DATE, b.birth) / 30, 1),
    w.distance_date = DATEDIFF(CURRENT_DATE, w.imm_date),
    w.house = b.house_name,
    w.hurdle_name = b.hurdle_name;
```

> 每日更新草株的生长月数、距离天数、所在站点和地块信息。

**query5 — 清理已淘汰草株的预警**：

```sql
DELETE w
FROM w_information_immunizationMessageinfo AS w
INNER JOIN basic_basicinfo AS b ON w.basic_id = b.id
WHERE b.state != 1;
```

> `basic_basicinfo.state != 1` 表示草株已淘汰、死亡或出售，对应预警记录不再有意义，予以删除。

**query6 — 清理"仅防护一次"的已完成记录**：

```sql
DELETE w
FROM w_information_immunizationMessageinfo AS w
INNER JOIN d_plantcare_immunizationinfo AS d
  ON d.basic_id = w.basic_id AND w.vaccine_id = d.vaccine_id
INNER JOIN threshold_setting AS t ON t.vaccine_id = w.vaccine_id
WHERE t.ifyear = 1
  AND d.imm_date IS NOT NULL;
```

> 对于标记为"全周期仅防护一次"（`ifyear = 1`）的防护剂，如果该草株已经有防护记录（`imm_date IS NOT NULL`），则删除对应的预警，不再提醒。

#### 3.6.3 事务管理

整个更新过程包裹在一个数据库事务中：

```python
try:
    db.session.begin()          # 开启事务
    # ... 执行所有 SQL ...
    db.session.commit()         # 全部成功则提交
except Exception as e:
    db.session.rollback()       # 任一失败则回滚
    db.session.flush()
    return jsonify({"code": 500, "msg": f"更新预警信息失败 {str(e)}"})
```

### 3.7 定时自动更新（task.py）

除了手动触发，系统还支持**每日凌晨 2:00 自动更新**预警数据：

```python
from apscheduler.schedulers.background import BackgroundScheduler
from apscheduler.triggers.cron import CronTrigger

def init_scheduler(app):
    scheduler = BackgroundScheduler()
    scheduler.add_job(
        func=lambda: update_database(app),
        trigger=CronTrigger(hour=2, minute=0),  # 每天凌晨 2:00 执行
        id='update_task',
        name='定时更新病虫害防护预警任务',
        replace_existing=True
    )
    scheduler.start()
```

定时任务的执行逻辑与手动更新接口 `update_thresholdsetMessageinfo()` 完全一致，确保预警数据每日自动刷新。

## 六、关键技术点总结

### 6.1 预警日期计算算法

预警模块的核心算法是**基于最近一次防护日期和防护周期阈值，计算下一次防护的截止日期**：

```
if 有防护记录:
    imm_date = MAX(imm_date)  from d_plantcare_immunizationinfo
    dead_date = imm_date + threshold
else:
    imm_date = birth
    dead_date = birth + threshold
```

`distance_date = 当前日期 - imm_date` 用于直观显示"距离上次防护已过去多少天"。

### 6.2 "循环防护" vs "仅防护一次"

| 模式 | ifyear | 行为 |
|------|--------|------|
| 循环防护 | 0 | 每次防护后，重新计算下一次截止日期；可多次提醒 |
| 仅防护一次 | 1 | 一旦记录防护完成，立即删除预警，终身不再提醒 |

### 6.3 性能优化

- **批量 SQL 执行**：更新时使用原生 SQL 而非 ORM 逐条插入，单次可处理数千条记录
- **事务包裹**：6 条核心 SQL 在同一个事务中执行，保证数据一致性
- **分页查询**：前端使用 `pageNum + pageSize` 分页，避免大数据量一次性加载
- **增量更新**：仅对 `ifchange == 1` 的防护剂执行 DELETE+INSERT 初始化，其他防护剂只做 UPDATE

### 6.4 空值保护

在预警列表查询中，对于 `ele_num`、`pre_num`、`cname` 等模糊搜索字段，后端增加了空值保护：

```python
basic = BasicBasicinfo.query.filter(
    BasicBasicinfo.ele_num.like(f'%{value}%')).first()
if basic:
    conditions.append(column == basic.id)
else:
    conditions.append(column == -1)  # 查无结果时返回空列表，不抛异常
```
